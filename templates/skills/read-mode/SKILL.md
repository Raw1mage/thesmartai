# Skill: read-mode

## 目的

讓 agent 在面向人類輸出**長篇說明性資訊**時，改用「分段閱讀 + 互動推進」模式，而不是一次傾倒整篇文字。

## 何時使用

在以下情況主動啟用 read-mode：

- 正在**回答使用者提問**，且答案屬於解釋型 / 教學型 / 分析型內容
- 預期回答會超過一個畫面，或明顯包含 3 個以上段落 / 小節
- 內容需要人類逐步理解，例如：
  - 架構解釋
  - 設計權衡
  - 長篇 debug / RCA 說明
  - 規格導讀
  - 教學式分解

## 何時不要使用

以下情況**不要**進入 read-mode：

- tool call / build / test / git / runtime 狀態回報
- 錯誤摘要、驗證結果、commit 結果、diff 結論
- 使用者要求「直接講完 / 一次講完 / 給完整答案」
- 答案本來就很短（1~2 小段可講完）
- 使用者明確要求純列表、純指令、純結論

## 操作規則

1. 先將內容拆成自然段落或章節。
2. 一次只輸出一段，不要一次貼完整長文。
3. 每段結尾停下，使用 `question` 讓使用者決定下一步。
4. 除非使用者要求一次講完，否則不要自動連續輸出後續段落。
5. 若內容中有結論可先講，第一段可先給「總覽 + 本段重點」，但仍須停下等待選擇。

## 建議段落大小

- 每段以 **3~7 個重點句 / 一個自然小節** 為原則
- 以「人類可在短時間內看完」為目標，而不是 token 最小化

## question 選項模板

預設優先提供以下選項：

1. `繼續下一段`
2. `這段有問題`
3. `先給摘要`
4. `直接看結論`

視內容可加：

- `給例子`
- `看程式層細節`
- `跳到實作建議`

## 啟用判斷 heuristics

若同時滿足以下兩點，應主動考慮啟用：

1. **回答類型是說明性資訊**，而非執行結果回報
2. **答案預期偏長**，或需多步理解

簡單判斷句：

> 如果現在這則回覆的主要內容是「幫人理解」，而不是「回報工具結果」，就優先考慮 read-mode。

## 退出條件

當出現以下任一條件時，退出 read-mode：

- 使用者選擇 `直接看結論`
- 使用者明確說「一次講完」
- 後續回合轉為命令執行 / 測試 / git / runtime 操作

## 目標效果

- 讓人類能以自己的閱讀速度推進資訊
- 減少一口氣輸出多頁說明造成的認知負擔
- 保持 technical explanation 的可讀性與互動性
